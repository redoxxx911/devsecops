# Final DevSecOps Report

## 1. Project Presentation

This project presents a complete DevSecOps delivery chain for a Zero-Trust Vault application. The application is composed of a FastAPI backend, a React frontend, an ELK monitoring stack, and a SecOps automation bot connected to GitLab, Elasticsearch, Telegram, and an LLM provider.

The objective is to demonstrate how security can be integrated into each phase of software delivery: code validation, static analysis, container build, image scanning, dynamic application testing, deployment, monitoring, and automated incident assistance.

## 2. DevSecOps Architecture

The architecture is organized around the following components:

| Layer | Component | Role |
| --- | --- | --- |
| Application | FastAPI backend | Provides authentication and protected secret retrieval endpoints. |
| Application | React frontend | Provides the user interface for login and secure data access. |
| CI/CD | GitLab CI/CD | Orchestrates quality, security, build, scan, and deployment jobs. |
| SAST | SonarQube / SonarScanner | Detects code smells, bugs, vulnerabilities, and maintainability issues. |
| Container security | Trivy | Scans Docker images for high and critical vulnerabilities. |
| DAST | OWASP ZAP | Tests the running web application from an external attacker perspective. |
| Monitoring | Filebeat, Elasticsearch, Kibana | Collects, indexes, and visualizes runtime logs. |
| Automation | SecOps bot | Queries SIEM data and GitLab pipelines, then summarizes incidents. |
| Deployment | Docker Compose / Kubernetes | Runs the stack locally or in a cluster environment. |

## 3. CI/CD Pipeline Explanation

The global GitLab pipeline contains nine stages.

### 3.1 Install

The install stage prepares Python dependencies for the backend. It validates that the project dependency files are usable inside a clean CI environment.

### 3.2 Lint

The lint stage checks code quality. The backend is checked with `flake8`, while the frontend is checked through the existing `npm run lint` command. These jobs are allowed to fail during the learning phase so the pipeline can still demonstrate the complete DevSecOps chain.

### 3.3 Test

The test stage runs backend tests with `pytest` and exports coverage in Cobertura XML format. This makes the result visible in GitLab and allows coverage to become a future quality gate.

### 3.4 SAST

The SAST stage runs SonarScanner against the backend, frontend, and SecOps bot source code. Its goal is to identify insecure coding patterns, maintainability issues, and vulnerabilities before the application is packaged.

### 3.5 Build

The build stage creates Docker images for the backend, frontend, and SecOps bot. Images are saved as tar artifacts so the next stage can scan the exact images generated by the build.

### 3.6 Container Scan

The container scan stage uses Trivy to inspect the Docker image artifacts for high and critical vulnerabilities. This stage is essential because a clean source code scan does not guarantee that the base image or installed packages are safe.

### 3.7 Push

The push stage uploads images to the GitLab Container Registry when the pipeline runs on the `main` branch. This separates local validation from release publishing.

### 3.8 DAST

The DAST stage runs OWASP ZAP baseline scanning against the deployed frontend. It checks the application as an external client would see it, including HTTP headers, cookies, exposed routes, and passive scan alerts.

### 3.9 Deploy

The deployment stage documents how the final application can be launched with Docker Compose or Kubernetes. Docker Compose is used for local/staging demonstrations, while Kubernetes manifests are provided for cluster deployment.

## 5. Security Scan Result Analysis

The repository contains several findings that are useful for a DevSecOps demonstration.

### 5.1 SAST Findings

The backend contains a hardcoded JWT secret. In a production system, this must be moved to a secret manager or environment variable. Hardcoded secrets make token forgery easier if the repository is leaked.

The backend also uses permissive CORS with `allow_origins=["*"]`. This is convenient in development but should be restricted to approved frontend origins in production.

The authentication endpoint contains demo credentials. This is acceptable for a controlled classroom demonstration, but real credentials must be stored securely and verified with a proper identity provider or password hashing mechanism.

The frontend contains a hardcoded API URL. This should be configured through environment variables per environment.

### 5.2 Container Scan Findings

Trivy is configured to report high and critical vulnerabilities in backend, frontend, and SecOps bot images. The expected remediation process is:

1. Update base images such as `python:3.11-slim`, `node:20-alpine`, and `nginx:stable-alpine`.
2. Update application dependencies in `requirements.txt` and `package-lock.json`.
3. Rebuild images and rerun the scan.
4. Document accepted risks when a vulnerability has no available fix.

### 5.3 DAST Findings

OWASP ZAP baseline scanning is configured for the running frontend. Typical findings may include missing security headers, missing cookie flags, exposed development endpoints, or weak browser protection headers. These findings should be reviewed and remediated through web server headers, frontend configuration, and backend response hardening.

### 5.4 Monitoring Findings

Filebeat collects container logs and forwards them to Elasticsearch. The SecOps bot searches for repeated `401` responses and error messages. This helps detect failed authentication spikes and application problems after deployment.

## 7. Deployment

### 7.1 Docker Compose

Docker Compose runs the backend, frontend, Elasticsearch, Kibana, Filebeat, and SecOps bot. It is the simplest deployment model for the project demonstration.

```bash
docker compose up -d
```

### 7.2 Kubernetes

The Kubernetes folder contains a namespace, deployments, services, and a kustomization file. It can be applied with:

```bash
kubectl apply -k kubernetes/
```

Sensitive values for the SecOps bot are represented as Kubernetes secrets and must be replaced before deployment.

## 8. Conclusion

This project demonstrates a complete DevSecOps workflow across the full software delivery lifecycle. The CI/CD pipeline validates the code, performs static analysis, builds Docker images, scans containers, runs dynamic security testing, and prepares deployment. The monitoring stack and SecOps bot extend the project beyond deployment by adding runtime visibility and automated incident assistance.

The main improvement areas are secret management, stricter CORS configuration, stronger authentication, environment-based frontend configuration, and enforcing pipeline gates once the project moves from demonstration mode to production mode.
